바이브 코딩이 뭔가요?

바이브 코딩(Vibe Coding)은 AI에게 자연어로 설명하면 코드를 만들어주는 방식입니다. 직접 코드를 한 줄 한 줄 치는 게 아니라, "로그인 페이지 만들어줘"처럼 말하면 AI가 코드를 생성합니다.

하지만 "코딩을 전혀 몰라도 된다"는 건 반쯤만 맞습니다. AI가 만든 결과물을 확인하고, 문제가 생겼을 때 어디가 잘못됐는지 파악하려면 기초 지식이 필요합니다.

이 가이드는 코딩 경험 0%인 사람이 AI 도구를 활용해 실제 서비스를 만들고 사업할 수 있게 되기까지의 현실적인 로드맵입니다.

Phase 1

기초 체력 기르기

2 — 4주

이 단계의 목표: AI가 만든 코드를 봤을 때 "이건 로그인 처리하는 부분이구나" 정도로 알아볼 수 있는 수준. 직접 코드를 술술 쓸 필요는 전혀 없습니다.

생활코딩에서 뭘 배워야 하나요?

생활코딩(opentutorials.org)은 한국어 무료 강의 중 입문자에게 가장 좋습니다. 전부 다 볼 필요 없습니다. 아래 순서대로만 보세요.

1 WEB1 — HTML & Internet

웹페이지가 어떻게 만들어지는지 배웁니다. 모든 웹 서비스의 기초입니다.

배우는 것

HTML 태그가 뭔지, 웹페이지가 어떻게 구성되는지, 텍스트/이미지/링크를 넣는 법

왜 필요한지

AI에게 "이 버튼을 왼쪽으로 옮겨줘"라고 말하려면, 화면이 어떤 요소들로 구성되는지 알아야 합니다.

어디까지 보면 되나요

끝까지 다 보세요. 짧고 쉽습니다. 하루면 충분합니다.

2 WEB2 — CSS

웹페이지를 예쁘게 꾸미는 법을 배웁니다. 색상, 크기, 배치 등.

배우는 것

글자 색 바꾸기, 배경 넣기, 요소 크기 조절, 가운데 정렬, 레이아웃 잡기

왜 필요한지

AI가 만든 화면이 마음에 안 들 때 "여백 좀 늘려줘", "이 버튼 색상을 파란색으로 바꿔줘" 같은 정확한 수정 요청을 하려면 CSS 개념을 알아야 합니다.

어디까지 보면 되나요

박스 모델, flexbox까지 보면 됩니다. grid는 건너뛰어도 괜찮습니다.

3 WEB2 — JavaScript

웹페이지에 동작을 추가하는 법을 배웁니다. 버튼 클릭, 데이터 처리 등.

배우는 것

변수 (값을 저장하는 상자), 조건문 (if — 만약 ~하면), 반복문 (같은 작업 반복), 함수 (작업 묶음)

왜 필요한지

모든 프로그래밍의 기본 개념입니다. AI 코드를 읽을 때 "여기서 로그인 성공하면 대시보드로 넘어가는 거구나"를 이해하려면 이 개념들을 알아야 합니다.

어디까지 보면 되나요

함수까지 보면 충분합니다. 객체, 클래스, 정규표현식은 나중에 필요할 때 보세요.

완벽히 이해 안 돼도 괜찮습니다. "이런 게 있구나" 수준이면 됩니다.

4 Git (버전 관리)

코드의 변경 이력을 관리하는 도구입니다. "세이브 포인트"라고 생각하면 됩니다.

배우는 것

저장(commit), 업로드(push), 다운로드(pull), 되돌리기(revert)

왜 필요한지

AI가 코드를 잘못 고쳐서 사이트가 망가졌을 때, Git이 있으면 이전 상태로 바로 되돌릴 수 있습니다. 없으면 처음부터 다시 만들어야 합니다.

추천

생활코딩 Git 수업을 보거나, GitHub Desktop 앱을 설치하면 클릭만으로 할 수 있습니다.

이 단계에서 하지 말아야 할 것

  • 코딩 문제 풀기 — 알고리즘 문제는 바이브 코딩에 불필요
  • 프레임워크 공부 — React, Next.js는 아직 안 봐도 됨
  • 완벽히 이해하려고 하기 — 70% 이해하면 다음 단계로
생활코딩 WEB1
HTML 기초 (하루면 끝)
무료 · 한국어
생활코딩 WEB2 CSS
꾸미기 (2-3일)

Phase 2

AI 도구 숙달

2 — 3주

이 단계의 목표: AI에게 말로 설명하면 원하는 기능이 만들어지고, 에러가 나면 스스로 해결할 수 있는 상태

AI-Native 개발이란?

전통적 코딩은 "문법 외우기 → 코드 직접 치기"였지만, AI-Native 개발은 다릅니다:

  • 코드를 쓰는 사람이 아니라 코드를 지시하는 사람이 됩니다
  • 중요한 건 프로그래밍 문법이 아니라 "뭘 만들고 싶은지" 명확히 설명하는 능력
  • AI가 만든 결과물을 검수하고 방향을 잡아주는 것이 핵심 역할
1 AI 도구 설치하고 써보기

Claude Code (터미널 기반)

터미널에서 AI와 대화하며 코드를 만듭니다. 가장 강력하지만 터미널에 익숙해야 합니다.

Cursor (에디터 기반)

VS Code와 비슷한 에디터인데, AI가 내장되어 있습니다. 처음 시작하기에 좋습니다.

처음엔 뭘 쓸까?

Cursor로 시작하는 걸 추천합니다. 화면이 있어서 직관적이고, 코드가 어디에 만들어지는지 바로 볼 수 있습니다. 익숙해지면 Claude Code로 넘어가세요.

2 프롬프팅 — AI에게 잘 시키는 법

바이브 코딩의 핵심은 "어떻게 시키느냐"입니다. 같은 AI라도 시키는 방법에 따라 결과가 천지차이입니다.

원칙 1: 구체적으로 설명한다

"로그인 만들어줘" (X) → "이메일과 비밀번호 입력 필드가 있는 로그인 페이지. 로그인 버튼 클릭 시 /dashboard로 이동" (O)

원칙 2: 한 번에 하나씩

한꺼번에 10가지를 시키면 품질이 떨어집니다. 하나 완성 → 확인 → 다음 작업.

원칙 3: 결과를 확인한다

AI가 만든 걸 무조건 믿지 마세요. 실행해보고, 눈으로 확인하고, 이상하면 수정 요청하세요.

원칙 4: 에러 메시지 = 보물

에러가 나면 당황하지 말고, 에러 메시지를 그대로 복사해서 AI에게 붙여넣으세요. AI가 알아서 고쳐줍니다.

좋은 프롬프트
"로그인 페이지를 만들어줘. 이메일과 비밀번호 입력 필드가 있고, 로그인 버튼을 누르면 /dashboard로 이동해. Supabase Auth를 써줘. 모바일에서도 잘 보이게 해줘."
나쁜 프롬프트
"로그인 만들어줘"
3 핵심 기술 스택 이해하기

직접 공부할 필요 없습니다. "이게 뭐하는 건지"만 알면 됩니다. AI가 코드는 알아서 씁니다.

4 AI-Native 학습법

전통적 학습과 AI-Native 학습은 완전히 다릅니다:

전통적AI-Native
문법 암기필요한 것만 AI에게 물어보기
바닥부터 코딩"이거 만들어줘"로 시작
에러 직접 디버깅에러 메시지 복붙하면 AI가 해결
문서 읽기AI에게 "이거 뭐야?" 물어보기
6개월 후 첫 프로젝트첫 날부터 뭔가 만들기

핵심 루틴: 만들고 싶은 거 생각 → AI에게 시킴 → 안 되면 에러 보여줌 → 고쳐달라 함 → 반복. 이게 전부입니다.

Phase 3

실전 프로젝트

4 — 6주

이 단계의 목표: 하나의 제품을 처음부터 배포까지 완성해본 경험. 이게 있으면 뭐든 만들 수 있다는 자신감이 생깁니다.
1 포트폴리오 사이트 만들기 (첫 번째 프로젝트)

자기소개 페이지를 만들어서 인터넷에 올려봅니다. 가장 쉽고, 배포까지 경험할 수 있습니다.

AI에게 이렇게 시키세요: "Next.js로 개인 포트폴리오 사이트 만들어줘. 자기소개, 프로젝트 목록, 연락처 섹션이 있으면 돼."

2 CRUD 앱 만들기 (두 번째 프로젝트)

메모장이나 할일 목록처럼 데이터를 만들고(Create), 읽고(Read), 수정하고(Update), 삭제하는(Delete) 앱을 만들어봅니다.

여기서 데이터베이스 연동을 처음 경험합니다. 거의 모든 서비스의 기본 구조가 CRUD입니다.

3 사업 아이템 MVP (세 번째 프로젝트)

실제로 만들고 싶은 서비스의 최소 버전을 만들어봅니다.

MVP(Minimum Viable Product) = 핵심 기능 딱 1개만 넣은 버전. 예를 들어 배달앱을 만들고 싶다면, 첫 MVP는 "메뉴 보기 + 주문하기" 딱 이것만.

이 단계에서 배우는 것

  • AI 코드가 동작하지 않을 때 대처하는 법
  • 머릿속 아이디어를 구체적 지시로 변환하는 법
  • 배포 → 버그 발견 → 수정 사이클 반복
  • "이건 AI한테 시키고, 이건 직접 해야 한다"를 구분하는 감각
Phase 4

사업 적용

실전

이 단계의 목표: AI를 활용해 실제 서비스를 만들고 수익을 내는 것

AI로 잘 되는 것

  • MVP 빠르게 만들기 기존 몇 달 → 며칠
  • 랜딩 페이지, 관리자 페이지 같은 정형화된 것
  • API 연동, 데이터 처리 기능
  • 디자인 수정, 반응형 적용

직접 챙겨야 하는 것

  • 보안 — AI 코드에 보안 허점이 있을 수 있음 중요
  • 결제 로직 — 돈 관련은 반드시 전문가 검증
  • 법적 요건 — 개인정보보호법, 이용약관, 사업자 등록
  • 스케일링 — 사용자 수천 명 넘어가면 구조 변경 필요

사업 관점 핵심 3가지

  • 코드보다 문제 정의가 더 중요 — "뭘 만들지"가 "어떻게 만들지"보다 우선
  • MVP는 핵심 기능 1개만 넣고 시장 반응 먼저 확인
  • 기술 공부에 빠지지 말 것 — 만드는 데 필요한 만큼만 배우기

전체 흐름 요약
